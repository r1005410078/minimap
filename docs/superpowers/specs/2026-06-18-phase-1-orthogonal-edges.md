# Phase 1 正交连线 Spec

> 对应 [ROADMAP.md](../../../ROADMAP.md) 第一阶段新增的"正交连线"功能项；是 Canvas 渲染器、Vue 组件壳两个切片之后的第四个切片，排在"布局切换动画"切片之前。
> 依赖 [Canvas 渲染器 spec](2026-06-18-phase-1-canvas-renderer.md) 的 `resolveEdges`/`drawEdge`/`renderScene`，[逻辑层 spec](2026-06-18-phase-1-core-logic.md) 的 `layout.nodes`/`layout.groups` 世界坐标包围盒。

## 头脑风暴决策记录

- **范围**：正交折线（直角连线）应用到**所有**连线，包括父子树连线和 `graph.edges` 业务关系线，不只是父子树连线。
- **路由算法**：通用算法，不是只处理"父在左子在右"的简单情形——按主轴方向（左右布局看 x，上下布局看 y）判断 `from`/`to` 包围盒在主轴上的区间是否重叠；不重叠时按主轴方向选朝向对方的边出/入，转折点取两条朝向边坐标的中点；区间重叠（比如同深度兄弟节点之间的业务线，如示例图 `cluster-8→cluster-25`）退化成按交叉轴方向出/入。父子树连线因为同一父节点的子节点共享同一深度（同一主轴坐标），转折点天然对齐，形成公共"脊线"，不需要特殊处理。
- **终点画箭头**，方向跟路径最后一段一致。
- **数据契约**：`resolveEdges` 现有的 `from`/`to`（中心点）保持不变，新增 `fromBox`/`toBox` 两个包围盒字段供默认绘制使用；自定义 `edgeRenderer` 钩子本切片不暴露包围盒，仍然只拿到 `{from, to}` 中心点（和 Canvas 渲染器切片的既有契约一致）。
- **范围拆分**：本切片只做连线视觉样式（正交折线 + 箭头），不做布局切换动画、不做视口锚点补偿、不做演示页方向切换按钮——这三项留给下一个切片。先让折线本身稳定下来再加动画插值，避免一次改两个还在变化的东西。

## 模块 API 契约

### `src/minimap/orthogonal.js`

```js
orthogonalPath(fromBox, toBox, mainAxis) -> [{x,y}, {x,y}, {x,y}, {x,y}]
```

- `fromBox`/`toBox`：世界坐标包围盒 `{x, y, width, height}`（和 `layout.nodes`/`layout.groups` 里的条目同构）。
- `mainAxis`：`'x'`（左右布局）或 `'y'`（上下布局）。
- 返回固定 4 个点，对应三段折线：出发边中点 → 转折点1 → 转折点2 → 进入边中点。两个转折点共享同一个坐标（取决于走主轴分支还是交叉轴分支），所以中间那段在该轴上是一条直线。
- 算法：
  1. 算 `fromBox`、`toBox` 在主轴上的区间 `[start, start + extent]`，判断是否重叠（区间端点相等，即两者恰好相切，按重叠处理——因为相切意味着主轴方向上没有可以走折线的安全间隙）。
  2. **不重叠**：按两者主轴中心坐标谁更靠前，选 `fromBox` 朝向 `toBox` 的那条边（取该边在交叉轴上的中点）作为出发点，`toBox` 朝向 `fromBox` 的那条边（交叉轴中点）作为到达点；两个转折点的主轴坐标 = 两条朝向边主轴坐标的中点，交叉轴坐标分别等于出发点/到达点自己的交叉轴坐标。
  3. **重叠**：同样的逻辑换到交叉轴上做——出发点/到达点改成按交叉轴方向选朝向对方的边，转折点坐标在交叉轴上取中点。
  4. 同一父节点的所有子节点共享同一个 `fromBox`，且它们的 `toBox` 们在主轴上的坐标也相同（层级树布局的性质），所以这些子节点边的转折点主轴坐标都一样，视觉上对齐成一条公共脊线。

### `src/minimap/renderer.js` 改动

- `resolveEdges(graph, layout)` 返回的每条边对象增加 `fromBox`/`toBox`：解析端点时除了算中心点（现有逻辑不变），同时记下解析到的包围盒（节点包围盒或分组框包围盒，与中心点来自同一个盒子）。
- 默认绘制改成：拿 `fromBox`/`toBox`/`mainAxis` 调 `orthogonalPath` 得到四个点，`beginPath` + 三段 `lineTo` + `stroke`；再在最后一段的方向上画一个箭头三角形（用 `theme.edge.color` 填充，大小用 `theme.edge.arrowSize`）。
- `renderScene` 需要知道 `mainAxis`（左右布局对应 `'x'`，上下布局对应 `'y'`）——具体怎么把 `direction` 透传到 `renderScene`/默认绘制函数，由 plan 阶段定（`scene.layout` 目前不带 `direction` 字段，需要从 `scene` 的其他地方传入，比如新增 `scene.direction` 或复用 `viewport`/已有参数）。
- 自定义 `edgeRenderer(ctx, params)` 的 `params` 形状不变：`{ edge, from, to, theme, viewport }`，`from`/`to` 仍是中心点屏幕坐标，不带 box。

### `src/minimap/theme.js` 改动

- `theme.edge` 增加 `arrowSize`（像素，默认 6）。

## 验收标准

- 左右布局下，父子树连线画成"横-竖-横"折线，同一父节点的所有子节点的折线转折点 x 坐标相同（视觉上对齐成公共竖线）。
- 上下布局下，父子树连线画成"竖-横-竖"折线，同一父节点的所有子节点的折线转折点 y 坐标相同。
- 业务关系线（`graph.edges`）也画成折线 + 箭头，包括两端在主轴上区间重叠的情形（如示例图 `cluster-8→cluster-25`），不出现来回折返的异常路径。
- 折线终点有箭头，方向跟最后一段路径方向一致。
- 传入自定义 `edgeRenderer` 后，默认折线绘制不再发生，自定义函数仍然拿到中心点 `from`/`to`（契约不变）。
- 既有测试（绘制顺序、视口裁剪、压力图、折叠端点路由）保持通过，不回归。

## 测试清单

- `orthogonalPath`：主轴不重叠（左右布局下目标在右/在左两种）、主轴区间重叠退化交叉轴（左右布局）、上下布局的对应情形——覆盖 4-6 个用例。
- 同一父节点多个子节点：取两个子节点的路径，断言转折点坐标一致（验证"公共脊线"性质）。
- `renderer.js`：mock ctx 断言折线绘制调用（每条边发出 3 段 `lineTo` 而不是 1 段）、箭头三角形的绘制调用存在；自定义 `edgeRenderer` 替换默认绘制时仍然只拿中心点，不拿 box。
- 压力图（10000 节点）：复用现有 `stats.drawn`/`durationMs` 断言模式确认折线绘制不显著拖慢渲染，不新增专门的性能测试。

## 风险与取舍

- 主轴重叠判断用"区间是否重叠"而不是"中心点距离小于阈值"，避免给不同大小的包围盒选错阈值；区间相切（端点相等）按重叠处理，理由见上方算法第 1 步。
- 这套路由算法不做避障（不检测折线是否穿过其他节点），对复杂的业务关系线布局可能出现折线穿过无关节点的情况；当前示例图和压力图数据不会触发明显的穿越问题，更完整的避障留给未来如果真的需要再做。
